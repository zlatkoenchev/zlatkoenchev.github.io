BMS

I start to work on this project because of package of e.g. 80 Li-Ion 21700 cells which got by occasion. Unfortunately I cannot find good BMS (good means to be able to set min/max battery voltage (I want to charge to e.g.4.0...4.1V instead 4.2V) low consumption, balancing to work all the time)

I decide to use AVR AtTiny13 MCU for cell node. Currently only FW is ready (but not tested on real HW). I plan to use UART on 9600bps with autobaudrate adjustment to compensate AVR clock jitter.

There should be one main cell node (probably PIC based) which to control charge, discharge and to communicate with other cell nodes.

I bought several DC-DC inverters 24->5V RTC2-2405SRW-R which I want to use for pack to cell charging. They can deliver 400mA@5V, but if connected to 3..4V cell they provide up to 700mA but get overloaded. I increase their current limiting resistor form 0.56 to 1Ohm and now with connected cell they deliver 400mA (I want to use them as current source). They will be controlled via opto MOSFET switches and the 5V output will be connected via diode to each cell.

 

HOME Automation system

I decide to to create my own system after analyze of commercial ones (Z-wave, Zigbee, Wifi, etc). Most of them needs hub or Internet connection to work and consume so much power, that you have to stop you water heater for 1..2 days every month to compensate the self consumption!

My system is standalone, no hub is required and consumes just 0.1W (1.5VA) per node powered from 220V. I have also kinetic powered switches, CR2045 battery powered sensors for windows and last one is TEG thermo valve (powered by thermal generator). I have 3 nodes (HW):

-       “Thermostat” – node with display and 2 buttons, digital PIR sensor, I2C Temp&Hum sensors, RTC (only node one hase Real Time Clock and broadcasts the time to the other)

-       “Actuator” – node with Bistable relay 16A or MOSFET dimmer or Triac, supports I2C Temp&Hum sensor, DS1820 temp sensor, digital PIR

-       “Switch” – node which only transmit switch state – window sensor, light switch, remote control. Some are powered by ZF electromagnetic generator (no batteries)

Communication protocol is broadcast based, signal oriented with self describing signals (each signal has ID and length transmitted)– i.e. adding new signals do not require to change SW in all nodes – they simply will skip unknown signals. Nodes are subscribed to specific signals from specific nodes.

All nodes except “Switch” type have bootloader and SW can be updated over the radio at any time. I have created PC USB-RF gateway based on Olimex PIC board and RF boards. I have PC SW written in C# which allows me to reflash devices and to configure them and monitor network if needed.

I am using several “Actuator” to control lights (as their consumption is less than Chinese PIR sensors more than 2 times) as well some other loads like floor heating. Window sensors prevent heating if window is open.

220V nodes are powered with capacitor power supply with 100nF capacitor which provides 6mA average current, but it is enough to drive 12V bistable relay (220uF buffer capacitor provides energy for 20ms activation). This type of supply is very simple and reliable – you can connect it even to 600V AC and it will work (X2 cap is used), while any pulse supply will explode.

UPS for home lights

The idea for this project popup from a storm – the electricity of the house of my wife stops and we start to search for devices like phones and tablets which have backlight in order to move in the dark…We have super modern phones or tablets, but we stay in the dark? I decide to solve the problem. First I think that switching to 12V DC power for lights is the best option, because the battery is just connected the lines and that’s all. But after some thinking and calculations, I decide to use 220V UPS because efficiency of 220V LED lamps reach already 160…200lm/W, while 12V LED lamps stay around or below 100lm/W. Additionally 12DC supply has to work all the time and should be extremely reliable.

I check dozens of standard UPS and invertors, but unfortunately I did not find my. Almost all have no load consumption of 300…400mA. This means that if the power stops early in the morning, the UPS will draw 4Ah for next 10h and when night comes and we need light, the battery will be almost empty (I want to use standard 12V / 7Ah battery). Finally I found in one PV inverter spec. description of so called “search” mode which can decrease consumption from 400 to 40mA. This was what I need! But the inverter was quire big and powerful and I need only 50-100W max. I have asked one local Bulgarian manufacturer of the UPS to add such function in their products and to buy one, but their answer was “not possible”. Ok, I buy one toroidal 100VA 220V-9V transformer, one VNH5019 H-bridge board and with old giant 40 pin PIC, I have created sine UPS. After some weeks experiments I have UPS with search mode which consumes below 2mA!!! Search mode generates one sine wave every 2 seconds and checks for load change. When there is no 220V power, after max 2 sec. the inverter goes on and you have light again! If you switch off lights, the UPS immediately enters search mode. Thus it can work hundreds of hours without significant battery discharge and to provide you power when you need! First prototype works 1 year and I replace it with second one with specially designed PCB and discrete MOSFET H-bridge which more efficient than VNH5019 (I like the efficiency). For stabilize the output voltage I use optocoupler, but it turns out that it has high temperature dependency (e.g. 1V/C). My last version is not using voltage feedback but only battery current supply feedback – I compensate sine PWM using transformer equations (knowing transformer winding resistance) to keep output stable – and it works better than optocoupler. Charging of the battery is realized with H brigde too, by driving low side MOSFETS only. So the final schematic is quite simple.

 

Technical data of UPS for lights:

100VA transformer

Pure Sine Wave

120mA no load current

<2mA search mode current

2 sec search period
